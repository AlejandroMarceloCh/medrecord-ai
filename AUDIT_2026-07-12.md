# Auditoría exhaustiva — 2026-07-12

Cuatro auditores en paralelo sobre el árbol completo: diseño/UX, backend/seguridad, producto/piloto,
y cruce con el material de los cursos UTEC (DPD + BD2). Todo verificado leyendo el código real.

**Veredicto: el producto NO está listo para tener un paciente delante.** No por falta de features
—el human-in-the-loop, el cifrado, la firma inmutable y el consentimiento son mejores que los de
casi cualquier competidor— sino porque **el pipeline nunca se ejerció en condiciones de turno real**
y porque **el camino de despliegue que el propio repo recomienda corre sin autenticación**.

Los 69 tests que pasan verde no cubren nada de esto: los 12 suites inyectan sidecars con `fields`
fixture. Ninguno toca Whisper ni Ollama de verdad.

## P0 — bloquean el piloto

### 1. El deploy documentado corre SIN autenticación (fail-open)
`server.js:216` · `auth.js:30-36` · `.env.example` · `DEPLOY.md`

```js
const authRequired = () => auth.countUsers() > 0 || !!REQUIRED_TOKEN;
function canSee(identity, rec) { if (!identity) return true; ... }
```

`.env.example` no menciona `MEDRECORD_ADMIN_USER/PASS` ni `MEDRECORD_TOKEN`. DEPLOY.md dice
`npm start` + `cloudflared tunnel --url`. Sin usuarios y sin token, `authRequired()` es `false`,
el gate hace `next()` para todo, `canSee(null, rec)` devuelve `true` y el WS difunde `publicRec`
completo a cualquiera. **Cualquiera con la URL del túnel lista todas las historias, descarga el
audio descifrado, edita, firma y borra. Sin login.**

Segundo camino al mismo estado: `loadUsers()` hace `try { readEncrypted } catch {}` sin distinguir
"no existe el archivo" de "no puedo descifrarlo". Si la `.master.key` cambia o el disco falla, el
server arranca con 0 usuarios y desactiva la auth en silencio.

**Fix:** separar `ENOENT` de fallo de descifrado (si el archivo existe y no descifra → no arrancar).
Modo abierto solo con `MEDRECORD_OPEN=1` explícito; en producción sin usuarios ni token → `exit(1)`.
Documentar todas las variables en `.env.example`.

### 2. Una `.master.key` truncada se sobrescribe → pérdida irreversible de toda la data
`crypto.js:20-31`

```js
try {
  const raw = fs.readFileSync(p);
  if (raw.length >= 32) return raw.subarray(0, 32);
} catch { /* no existe → la creamos */ }
const key = crypto.randomBytes(32);
fs.writeFileSync(p, key, { mode: 0o600 });   // ← pisa el archivo corto
```

Si el archivo existe con menos de 32 bytes (copia interrumpida, `scp` a medias, restore cortado,
disco lleno), **no entra al catch**: genera una clave nueva y pisa la real. Todos los sidecars y
audios quedan matemáticamente irrecuperables, y `loadAll()` los renombra a `.corrupt`.
RESTORE.md pasos 3-4 mueven la clave a mano: es exactamente el escenario que produce el archivo corto.

**Fix:** si el archivo existe y mide menos de 32 bytes → `throw`, nunca regenerar. Generar solo con
`ENOENT`. Test de regresión: escribir 10 bytes, arrancar, debe fallar.

### 3. Grabación fantasma: el cronómetro y la onda no están conectados al micrófono
`mobile.html:283-286` (setInterval atado a `phase`) · `mobile.html:185-198` (`@keyframes mr-wave`, CSS puro)

No hay `AnalyserNode`, no hay `rec.onerror`, no hay `track.onended`. Entra una llamada al celular,
iOS le quita el micrófono, y la pantalla **sigue diciendo "Grabando 14:32" con la onda animándose**.
El médico termina la consulta convencido de que grabó.

Es el peor fallo posible: silencioso, invisible, y destruye la confianza el día 1. Una onda que
miente es peor que ninguna onda.

**Fix:** nivel real con `AnalyserNode` + RMS por frame. `rec.onerror` y `track.onended` → parar y
avisar. Umbral de silencio sostenido → banner "No estamos captando sonido".

### 4. El móvil no arranca sin internet — y su promesa central es la cola offline
`mobile.html:77-79`

Carga React **development** + `@babel/standalone` (≈3 MB) desde `unpkg.com` y transpila JSX en el
navegador. `build.mjs` solo **copia** `mobile.html` a `dist/`, así que producción también depende del
CDN. No hay service worker en todo el repo.

La app le dice al médico *"Sin conexión. Tus grabaciones se guardan y se suben solas al volver"*
(`mobile.html:524`), pero si abre la app sin señal, **la app no existe**: pantalla en blanco.

**Fix:** bundlear el móvil con esbuild igual que la web, fuentes locales, service worker que cachee
el app shell.

### 5. Sin cola: los audios se procesan en paralelo
`server.js:294`

`processRecording(rec)` se dispara directo desde el handler del POST. Whisper corre con
`-t <nº de CPUs>` y carga large-v3 (3.1 GB) **por proceso**. El propio código admite la invariante
que viola: `loadAll()` sí serializa el resume, con el comentario *"re-disparar todas a la vez tras un
crash satura Whisper/Ollama. El resto del pipeline asume 'una a la vez'"*. El POST no lo asume.

En emergencia: 5 audios en 3 minutos → 5 procesos × 3.1 GB + Ollama (4.7 GB residente por
`keep_alive:30m`) en una Mac de 16 GB → swap, cada job 3-5× más lento, y ahí pega el siguiente P0.

**Fix:** cola FIFO con concurrencia 1 (un array + un flag `busy`, ~10 líneas) + estado `queued` con
posición visible.

### 6. Timeout de Whisper fijo, no por duración
`whisper.js:27` — `WHISPER_TIMEOUT_MS = 20 min` de reloj de pared, independiente de si el audio dura
3 minutos o 40. Al vencer: `SIGKILL` y error genérico.

Combinado con el P0 anterior: un audio de 20 min tarda más de 20 min de pared → se mata → el médico
toca Reintentar → vuelve a competir → vuelve a fallar. Cascada.

**Fix:** `timeout = max(5min, durationSec × K)`. `durationSec` ya llega del móvil y hoy solo se usa
para mostrar.

### 7. `num_ctx: 8192` fijo → truncado silencioso de la anamnesis
`llm.js:134`

Una consulta de 20 min son ~2,500-3,500 palabras ≈ 4.5-6k tokens, más el system (~500) y el JSON de
salida (~600). Está **en el borde**; 30 minutos lo pasa. Ollama trunca **descartando tokens del
inicio** — o sea se come el SYSTEM con el esquema, o el principio del audio, que es justo donde están
filiación y motivo de consulta.

`normalize()` rellena lo que falta con `''`, **indistinguible de "no se mencionó"**.

**Fix:** medir la longitud del transcript y subir `num_ctx` dinámicamente (qwen2.5 soporta 32k), o
trocear por secciones. Mínimo: si excede el umbral, `rec.fieldsError = 'transcripción demasiado
larga: revisa manualmente'`.

### 8. Números críticos (presión arterial, dosis) sin validación contra la transcripción
`llm.js:62-78` · `clinical.jsx:31-32`

`buildSources()` descarta las citas que no son verbatim — excelente. **Pero el campo se muestra igual
aunque su cita se haya descartado.** En la UI, "campo con evidencia verificada" y "campo sin ninguna
evidencia" se distinguen **solo por un punto teal de 5 px**.

Escenario: el médico dice "está en 140 sobre 90", Whisper transcribe mal, el LLM emite
`presion_arterial: "120/80"` sin fuente válida. El médico confirma en cadena y firma una presión
inventada.

**Fix (el de mayor retorno de toda la auditoría):** validación numérica en el server post-LLM. Para
PA/FC/T°/SpO₂ y dosis, comprobar que los dígitos aparecen **literalmente** en el transcript; si no,
vaciar el campo y marcarlo `sin_evidencia`. En la UI, reemplazar el punto sutil por una etiqueta
explícita **"SIN EVIDENCIA EN LA TRANSCRIPCIÓN"**.

### 9. El error de carga se disfraza de estado vacío
`app.jsx:392` (la rama de `loadError` solo aplica a `view==='workbench'`) · `listing.jsx:226-239`

Servidor caído o WS muerto → `recs=[]` → el médico lee *"Sin consultas en esta sección"* y cierra la
laptop creyendo que terminó el día. Confusión error/vacío que se traduce en consultas sin firmar.

**Fix:** pasar `loadError` a `ListingView`. Un vacío solo es legítimo cuando la carga **fue exitosa**
y devolvió 0.

### 10. No se puede cancelar ni descartar una grabación, en ninguna superficie
`mobile.html:494-499` (único control: Stop, y `onstop` encola y sube) · `clinical.jsx:159`
(`onDelete` se recibe como prop y **nunca se usa**; `handleDelete` en `app.jsx:305` es código muerto)

El paciente retira el consentimiento a mitad de consulta, o el médico arranca con el paciente
equivocado. La única salida es Stop → el audio se sube, se transcribe con un LLM y queda en el
servidor **para siempre**, sin UI para borrarlo. Es un problema clínico y de consentimiento.

**Fix:** botón "Descartar" en la pantalla de grabación con dialog destructivo. Cablear `onDelete` en
el workbench.

### 11. Sin arranque automático ni túnel estable
No hay LaunchAgent de MedRecord ni pm2. Si la Mac se reinicia por una actualización de macOS, **el
server no vuelve**. Y `cloudflared` **ni siquiera está instalado**, pese a que DEPLOY.md lo
recomienda; el túnel gratuito genera **una URL nueva en cada arranque** → el médico reinstalaría el
PWA todos los días y perdería el `localStorage` (token, config, diccionario) con cada cambio de origen.

**Fix:** LaunchAgent con `KeepAlive=true` + `caffeinate -dimsu`. Túnel: named tunnel con dominio
propio, o mejor **Tailscale** (MagicDNS + HTTPS de tailnet, el micrófono funciona, no expones nada a
internet público).

## P1 — daña la confianza

### Integridad de datos

- **El pipeline async pisa una historia YA firmada.** `server.js:507-539` — `/reextract` responde
  `{ok:true}` **antes** de llamar al LLM, y al aterrizar 30 s después no revalida `r.reviewed`. Si el
  médico firma mientras el LLM corre, los campos firmados se reemplazan por la salida cruda de la IA,
  `confirmed` se vacía, `status` vuelve a `'done'` y la firma queda apuntando a un contenido que ya no
  existe. El Sprint 11 cerró la guarda **de entrada**; la de salida no existe.
  **Fix:** capturar una generación al lanzar (`r.gen`) y revalidar al aplicar.

- **`persist()` se traga los fallos de escritura.** `server.js:43-52` — disco lleno o permisos → el
  `PUT` igual responde `200` con la firma. La UI dice "firmado". Al reiniciar, `loadAll()` lee el
  sidecar viejo: **la firma y las ediciones desaparecen**. Mismo patrón en `saveUsers` y en el
  `audit()` (un audit log que pierde entradas en silencio no sirve como evidencia legal).

- **El payload firmado no cubre el consentimiento ni la traza del human-in-the-loop.**
  `server.js:385-390` — quedan fuera del sello: `consent` (la base legal de todo el procesamiento),
  `confirmed` (qué campos atestó el médico) y `fields_ia` (la única evidencia de qué es humano y qué
  es máquina). Alguien con acceso al sidecar puede poner `consent.granted=true`, vaciar `confirmed`, o
  reescribir `fields_ia` — y `/verify` sigue diciendo `valid: true`.
  **Fix:** versionar la firma a `v:2` incluyendo esos tres campos.

- **`version` no avanza en retry/reextract/pipeline** → el optimistic locking no protege nada.
  Solo `PUT /fields` la incrementa. Lost update sin ningún 409.

- **`DELETE` no usa borrado seguro.** `server.js:542-551` usa `fs.rmSync` mientras la purga por
  retención usa `enc.secureDelete()`. El borrado explícito del médico —el que ejerce el derecho de
  supresión de la Ley 29733— deja el ciphertext recuperable del disco.

- **PHI en claro en `/tmp`.** `crypto.js:2` promete que "la data médica NUNCA queda en claro en
  disco", pero cada transcripción descifra el audio a `os.tmpdir()` y `whisper.js` escribe
  `out.txt` con **la transcripción completa en texto plano**. Ambos se borran con unlink, sin
  sobrescribir.

### Degradación

- **Si Whisper falla, el médico no puede escribir nada a mano.** `clinical.jsx:287-295` reemplaza
  **todo el formulario** por una pantalla de error. El audio quedó guardado y es reproducible, pero
  no hay forma de llenar la historia escuchándolo. En cambio, si Ollama cae, la degradación está
  **muy bien resuelta** (transcripción disponible, banner honesto, médico llena a mano con la
  evidencia al lado). Falta hacer lo mismo para Whisper.

- **`/health` existe pero la web nunca lo llama** (0 referencias en `src/web/` y en `mobile.html`).
  Si Ollama cae a mitad de turno, el médico se entera historia por historia.
  **Fix:** chip de salud en el header + banner global.

- **Sin `timeslice`**: `rec.start()` sin argumento → todo el audio vive en RAM hasta el stop. iOS
  Safari mata pestañas en segundo plano; 20 minutos de consulta es todo o nada. La cola de IndexedDB
  (muy bien hecha) solo protege **después** del stop.

### Flujo

- **Nombre obligatorio para poder grabar.** `mobile.html:279` — `canRecord` exige nombre + DNI válido
  + consentimiento, y el consentimiento se resetea tras cada grabación. Con 12-30 pacientes/día, el
  flujo real es "entra el paciente → grabo → después identifico". La web ya tolera pacientes sin
  nombre (`recName()` → "Sin identificar"). Exigir solo el consentimiento; nombre y DNI opcionales o
  capturables después.

- **Sin pausa, y sin forma de añadir un segundo audio al mismo paciente.** Entra la enfermera, el
  médico sale 2 minutos: hoy tiene que **parar** (se sube) y volver a grabar → dos historias distintas
  para un mismo paciente, sin forma de unirlas.

- **Identidad del móvil rota con auth activa.** `mobile.html:201` lee
  `localStorage['medrecord.token']` pero **no hay ninguna pantalla para setearlo**. Y si sube con el
  device token, `ownerId = null` → un médico no-admin **nunca ve las grabaciones que él mismo hizo**.
  Hoy funciona por accidente porque el único usuario es el admin.

- **Audio de 0 bytes se descarta en silencio.** `mobile.html:439` — `if (blob.size > 0) enqueue(...)`.
  Sin `else`. La UI vuelve al formulario limpio, idéntica a un éxito.

- **`Esc` descarta ediciones sin guardar.** `app.jsx:352` es el único camino de navegación que no
  llama `confirmLeave()`. El médico corrige el diagnóstico, aprieta Esc por reflejo, pierde la
  corrección sin un aviso.

### Seguridad (segunda línea)

- **Upload sin rate limit ni cola** → OOM. El throttle solo existe en `/login`.
- **Enumeración de usuarios por timing** (`auth.js:75-80`: si el usuario no existe, no corre scrypt) +
  **lockout dirigido** (5 intentos basura contra un username conocido lo bloquean indefinidamente).
- **El WS congela la identidad en el handshake**: logout y expiración no cortan el stream de PII. Sin
  `maxPayload`, sin cap de conexiones, sin validación de `Origin`.
- **CSRF/CSWSH**: la única defensa es `SameSite=Strict`, que **es ciego al puerto**. Cualquier otro
  servidor en `localhost` (tienes UTEC Gym en :8000, Vite en :5173) es *same-site* y puede firmar
  historias o abrir un `ws://localhost:3000` que sí lleva la cookie.
- **Prompt injection desde la transcripción**, y `_fuentes` da una falsa garantía: el atacante
  controla la transcripción, así que le basta citar una frase que él mismo dijo para que la cita
  verifique y la UI la resalte como "evidencia real". El human-in-the-loop es lo que salva.
- **`audioMime` viene del cliente y se refleja como `Content-Type`** → XSS almacenado si el admin
  abre la URL del audio.
- **Retención de audio desactivada por defecto** (`|| 0`) y no documentada. La promesa de "retención
  con borrado seguro" está apagada en cualquier instalación que siga el `.env.example`.
- **La misma clave sirve para AES-GCM y para el HMAC de firma.** Y el HMAC no da no-repudio: el
  servidor conoce la clave, así que un admin puede forjar la firma de cualquier médico.

### Diseño

- **El móvil ignora por completo el sistema de diseño.** `mobile.html:110` — acento
  `oklch(0.52 0.20 277)` = **índigo/violeta**, literalmente el ítem anti-slop *"purple sin pensar"*.
  `theme-color #4f46e5`, Hanken Grotesk + IBM Plex Mono, fondo gris frío. Nada de Clinical Precision
  llegó a la superficie que se usa con el paciente delante. Los dos manifests también.
- **Contraste: `--faint #A8A29E` = 2.33:1** (mínimo 4.5:1), y etiqueta **todos** los campos clínicos a
  10.5 px, más la fecha, la duración y el DNI del paciente. Riesgo de confundir campos en una pantalla
  de consultorio con reflejo.
- **La tabla densa decidida para "Por revisar" no está en el código**: es un `grid auto-fill
  minmax(280px)` de cards idénticas (~6 pacientes por pantalla vs 25 en tabla).
- **El vínculo campo→evidencia es solo-mouse** (`onMouseEnter`, sin `onFocus`). El médico que tabula
  entre campos nunca ve la evidencia — la feature que justifica confiar en la IA solo funciona si
  mueves el mouse.
- **Sin `prefers-reduced-motion` en móvil**, cards del listing no navegables por teclado, botones sin
  estado loading (`handleRetry` con `catch {}` = no-op absoluto), toast de error sin causa ni salida.
- **Terminología partida**: "Historial" ≠ "Revisadas" para el mismo destino; "consulta" vs
  "grabación" vs "Audios de esta sesión" vs "Historia clínica".
- **PII en `localStorage`**: nombre y DNI de los últimos 5 pacientes en texto plano en el celular, sin
  expiración ni forma de limpiarlos desde la UI.

## Lo que los cursos UTEC aportan (DPD + BD2)

El material da tres cosas realmente accionables. El resto es ruido académico y está descartado abajo.

### 1. Confianza por campo + cita al segundo del audio
*Tractable (U5_T2 s.10) + Selection Policy CONFIDENT/UNSURE (U4_T1 s.13-14) + "comprehensible
systems" (U10_T2 s.24)*

La UI de peritaje de Tractable muestra por cada parte del auto: `VISIBLE 100% | OUTCOME: Replace |
CONF: 99%`. El perito no revisa el auto: revisa **la tabla ordenada por confianza**.

En MedRecord: cada campo con `valor | confianza | evidencia`, y el orden por defecto **no** es el de
la historia clínica sino **confianza ascendente** — lo dudoso arriba.

La confianza no sale del modelo (los logprobs de un qwen 7B están mal calibrados): sale del
**desacuerdo de un ensemble barato**. Correr la extracción 2-3 veces con prompts distintos —uno
"extrae X", otro "¿se menciona X? cita la frase textual"—. Coinciden → verde. Divergen → **ámbar,
UNSURE**, sube al tope de la cola. Costo: 2-3× inferencia local, aceptable porque es post-consulta.

Y como Whisper da timestamps, el offset de la cita mapea al **segundo del audio**: click en el campo →
salta al momento exacto en que el paciente lo dijo. Eso es la explicabilidad (P9 del checklist ético:
un 7B es *opaque* e irremediablemente lo será, pero puede ser **comprehensible** — y la explicación no
es SHAP, **es la cita**) y es **la defensa legal ante una auditoría**.

### 2. Campos de baja confianza VACÍOS, no pre-rellenados
*"Prevent automation bias" (U10_T2 s.31) + Google Translate turco (U5_T2 s.31-33)*

Google no arregló el modelo cuando "o bir doktor" devolvía *"he is a doctor"*: **cambió la interfaz**
para mostrar las dos opciones. Costo de modelo: cero.

En MedRecord: cuando qwen tenga dos candidatos (penicilina vs penicilamina, 500mg vs 50mg), **no
elegir**. Mostrar ambos como chips y que el médico resuelva con un clic.

Y el corolario duro: los campos de baja confianza aparecen **vacíos**, con la sugerencia al costado y
un botón "aceptar". La firma queda bloqueada hasta que cada campo ámbar haya sido tocado. Es la
diferencia entre *"el médico es responsable"* y *"el médico es responsable pero no tuvo cómo saberlo"*.

**El riesgo real ya existe:** con 10-15 clics de "Confirmar" por paciente × 12 pacientes = ~150
clics/día, el médico aprende a clicar sin leer. Métrica del piloto: **% de historias firmadas en
menos de 20 segundos** — proxy directo de "firmó sin leer".

### 3. El costo asimétrico del error define la métrica del piloto
*Fraude de tarjeta (U5_T2 s.35): 99% de detección pero 500,000 legítimas bloqueadas, contra márgenes
de 0.5-3.5%*

La métrica del piloto **no es** "% de campos extraídos correctamente". Es la asimetría: un campo
**alucinado que el médico no detecta** es catastrófico; un campo **vacío que llena a mano** es barato.

Consecuencia: el prompt debe estar sesgado a **abstenerse**. Sin evidencia textual literal → `null`.
KPI: **tasa de alucinación = 0**, medida sobre una muestra auditada a mano.

### 4. Reglas deterministas para cifras y dosis; el LLM solo para lo narrativo
*"¿Una heurística funcionaría?" (U1_T2 s.33)*

Hay campos que **no necesitan LLM y no deberían pasar por él**: fecha/hora/médico salen del contexto
de la app; signos vitales y dosis son **regex + normalización** ("ciento veinte sobre ochenta" →
120/80); medicamentos son **match contra vademécum**, no generación libre. Reservar el LLM para lo
narrativo (motivo, enfermedad actual, plan). Esto elimina alucinaciones **justo donde matan**.

### 5. Versionar el "modelo" en cada sidecar firmado — trivial y urgente
*"El pipeline captura las dependencias de código y datos" (U4_T1 s.37)*

MedRecord no entrena, pero **sí tiene un modelo versionable**: `prompt + versión de qwen + versión de
Whisper + template`. Guardar en cada sidecar: `{whisper_model, llm_model, prompt_hash, app_version}`.

Sin esto, cuando cambies el prompt en agosto **no vas a poder explicar por qué una historia de marzo
se ve distinta de una de mayo** — y con firma inmutable eso es un problema legal, no estético.
Esfuerzo: una hora. **Tiene que estar antes de la primera historia del piloto**, porque esas son las
que se van a auditar.

### 6. El diagnóstico incómodo sobre el orden de los sprints
*Tabla de 5 fases (U1_T2 s.32): Prueba de Concepto → Prototype → **MVP Pilot** → Dejar listo el MVP →
Expandir*

Los 14 sprints (cifrado, multi-usuario, firma, consentimiento, retención) son **"Dejar listo el
MVP"** — ejecutados **sin haber corrido el MVP Pilot**. No es tiempo perdido (el compliance en salud
es precondición de venta), pero el orden se invirtió, y ahora el riesgo es construir SQLite, búsqueda
y escala sobre un supuesto que nadie validó.

**Y el piloto no puede consistir en que el médico diga "está bacán"** (s.18: *"no se limite a obtener
un asentimiento de cabeza"*). Hay que instrumentarlo **antes** de instalar nada:

- **Baseline con cronómetro, 1 semana sin la app**: minutos de documentación por consulta.
- Después: mismos minutos + ediciones por campo + % de historias firmadas sin editar + **consultas
  abandonadas** (grabó y nunca firmó — el indicador más honesto de que la app no sirve).
- **Un solo número de titular: minutos ahorrados por consulta.**
- **Dos checkpoints con criterio de PAUSA numérico, fijado ahora y no cuando lleguen los datos**
  (si no, se racionalizan). CP1 semana 2: ¿grabó ≥80% de sus consultas sin que se lo recordáramos?
  CP2 semana 8: ¿ahorro ≥ X min **y** ≥ Y% firmadas sin edición mayor **y** 0 alucinaciones?

Y el corolario de impacto (s.21): si ahorra 4 min por consulta pero **igual se queda 30 min al final
del día revisando historias acumuladas, el impacto es cero**. La revisión tiene que ocurrir **dentro
de la consulta** (mientras el paciente se viste), no en un batch nocturno. Eso exige que revisar tome
menos de 60 segundos — lo que a su vez justifica el orden por confianza y los campos ámbar.

### 7. El protocolo de dictado de cierre — cero código, la palanca más grande
Whisper.cpp **no diariza**. El transcript es un bloque plano, así que lo que el **paciente especula**
("creo que es dengue") puede terminar en `diagnosticos`.

Fix de producto: con el paciente delante, el médico dicta 20 segundos estructurados al final:
*"PA 120/80, FC 88, impresión: faringitis aguda, plan: amoxicilina 500 cada 8 horas por 7 días"*.
Eso convierte un problema de **comprensión de diálogo** (que Whisper no resuelve) en uno de
**transcripción de dictado** (que resuelve bien).

### 8. Búsqueda de historias: Postgres directo, NO SQLite, NO pgvector
*BD2/08 s.29-33 + BD2/11 s.22-23, 53*

El médico va a buscar **"hipertensión"** y **no va a encontrar** la historia donde escribió
**"hipertenso"**. Ese es el bug que mata la feature, y **es de stemming, no de performance**.
`to_tsvector('spanish', ...)` lo resuelve porque ambas stemmean al lexema `'hipertens'`.

Y eso fuerza una decisión **antes** de migrar: **SQLite no tiene `tsvector`**. Su FTS5 usa el stemmer
`porter`, que es **para inglés**. Se puede compilar un tokenizador Snowball español, pero no viene por
defecto. **Si la búsqueda va al roadmap (y va), sáltate SQLite y ve directo a Postgres.** Migrar
sidecars → SQLite → Postgres es hacer el trabajo doloroso dos veces.

Diseño concreto (10 líneas de SQL, el 90% del valor): un `tsvector` que concatene los campos **con
pesos** — `setweight(A)` = diagnóstico, `B` = medicación y alergias, `C` = motivo, `D` = transcripción
cruda — más un índice **GIN** y una **stoplist clínica propia** (`paciente`, `refiere`, `niega`,
`acude`, `normal` son las nuevas stop words: `idf ≈ 0`; `amiodarona` y `Guillain-Barré` son las que
discriminan).

**pgvector queda descartado, y el propio material lo demuestra.** El corpus es todo historias del
mismo médico y la misma especialidad: **semánticamente casi idénticas por construcción**. Los
embeddings van a estar todos cerca y el ranking por coseno va a ser ruido (*"Every point's
neighborhood is the same"*, s.22-23). Full-text con IDF discrimina **mejor** justo por la razón
opuesta: premia lo raro y castiga lo común. No es una concesión pragmática, es el resultado correcto.
**Regla de decisión: no tocar pgvector hasta >50,000 historias O hasta que el médico pida
explícitamente "casos similares".**

### 9. El pitch: es *wrapping* que se vende como *improving*
*U5_T5 s.42-52: de 315 ejecutivos MIT, 50% crea valor con improving, 33% con wrapping, solo 19% con
selling*

El pitch a la clínica **no es** "tenemos IA". Es **"su médico documenta en 3 minutos en vez de 12"**,
en soles. El molde es **Trinity Health** (s.49): un KPI operativo (tiempo de respuesta de enfermeras
−57%) encadenado a un outcome económico (menos caídas → menos costo).

La cadena de MedRecord: `minutos de documentación ↓ → consultas por turno ↑ → S/ facturados ↑`.
Y la advertencia clave del deck: **ahorrar 5 min/consulta no vale nada si el médico no redirige ese
margen.** Hay que elegir cuál vendes: (a) atiende 2-3 pacientes más por turno → el ROI es aritmética
y la licencia se paga con un día de uso; o (b) el médico se va a su casa a la hora → retención del
médico, menos cuantificable pero a veces vende mejor.

**Selling queda descartado por escrito.** Vender datos clínicos, aun anonimizados, mata el negocio
—y **ese "no" es el mejor argumento de venta**: *"sus datos nunca salen de su servidor; revenderlos
no es ni puede ser nuestro modelo de negocio"*.

Bonus: **CIE-10 automático es plata directa** — el código diagnóstico es lo que la clínica **factura
a la aseguradora**, hoy se hace a mano y mal. Pero hoy el campo lo **inventa un 7B sin catálogo**
(`llm.js:36`, "solo si son claros" no es una defensa): apagarlo en el piloto o poblarlo solo por
lookup contra tabla local.

### Descartado explícitamente
- **Feature store, model registry, reentrenamiento, drift estadístico** — no se entrena nada.
- **Adaptive UI que reordena campos según el uso** — vector de error médico y rompe la auditabilidad
  de la firma.
- **COMPAS / fairness de la predicción** — MedRecord **no predice ni decide**: transcribe y estructura.
- **A/B testing** — con N=1 médico es estadísticamente vacío. Usar pre/post con el médico como su
  propio control.
- **Implementar índice invertido propio (SPIMI)** — `GIN` de Postgres *es* el índice invertido.
- **Analytic profiles / scores por paciente** — es el V3. Nombrarlo en el pitch, no construirlo.
  *"Las organizaciones no fracasan por falta de casos de uso; fracasan porque tienen demasiados."*

## Orden de ataque

**Antes de tocar a un paciente:**
1. Cerrar el fail-open de auth y la regeneración de `.master.key` (P0-1, P0-2) — son 2 horas y son
   catastróficos.
2. Cola serializada + timeout por duración (P0-5, P0-6).
3. Móvil: `AnalyserNode` real + `onerror` + `timeslice` + bundle sin CDN + service worker
   (P0-3, P0-4).
4. `num_ctx` dinámico (P0-7) + validación numérica de vitales y dosis (P0-8).
5. Versionar el modelo en el sidecar (1 hora, y tiene que estar antes de la primera historia).
6. LaunchAgent + Tailscale (P0-11).

**Antes del día 2:**
Formulario editable en estado `error` de Whisper · chip de salud en el header · `loadError` en el
listing · descartar grabación · protocolo de dictado de cierre (cero código) · benchmark honesto de
latencia con un audio real de 20 min — **ese número es la promesa del producto y hoy no existe**.

**Instrumentación del piloto (antes de instalar):**
Baseline con cronómetro · métricas de edición por campo · consultas abandonadas · los dos checkpoints
con umbral de pausa escrito.

**Post-piloto:** entidad paciente y multi-consulta · receta estructurada · nota SOAP · búsqueda por
DNI y full-text · Postgres.

**No hacer ahora:** SQLite, paginación, FHIR, pino/métricas. El Sprint 15 completo es prematuro para
1 médico y 12 pacientes/día — salvo el timeout dinámico de Whisper, que está enterrado como último
bullet del sprint más lejano y en realidad es P0.
